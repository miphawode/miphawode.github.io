第四卷 · Harness 工程专章 · 第 16 章

Harness 的 9 层参考架构

检查清单

我能把这次任务在九层上各自的答案，写成一份 spec 吗？还是有几层从来没人想过？

① 不合格的任务，是被一个模板/分诊挡在门外，还是照单全收？

② agent 是先出计划等人确认，还是读完 issue 直接改文件？计划能追到验收标准和文件范围吗？

⑤ 审批点是写在权限层/工具层（硬卡），还是只写在 prompt 里（软求）？

⑥ 它在隔离环境里跑，还是直接动真仓库、挂着生产 secret？出网是 allowlist 还是放任？

⑦ "算完成"有可执行判据吗？eval 环境隔离干净了吗，还是基础设施噪声混在分数里？

⑧ 谁签收？发起人能不能自批自己这次 run 的 PR？

⑨ 失败 case 有没有喂回前几层（入 eval、入规则、改模板），还是存了就忘？

我有没有把九层当组织架构在堆？哪几层其实几行约定就够？哪条脚手架已经过期该拆？

自测

拿你最近一次真实的 agent run，对着九层逐层问"这层谁在管"。卡在第几层答不上来？那一层就是你现在风险漏得最多的地方。

如果现在把这次 run 的聊天历史清空，只留你落在文件里的 trace 和恢复 artifact，下一个人（或下一个 agent）接得住吗？接不住，说明你的 ⑨ 还活在会消失的地方。

你给某个任务族写的权限矩阵，是按风险/路径/命令分了级，还是一刀切"全部要批"或"全部放行"？前者是 harness，后两者一个累死人、一个迟早出事。
