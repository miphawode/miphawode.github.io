# 页面仓库 Agent 约束

这个文件是给 AI Agent 看的项目规则。后续修改本项目时，必须优先遵守这里的约束。

## 项目目标

本项目是一个静态书籍网页：

- 左侧是章节目录。
- 右侧是当前章节内容。
- 每章只展示“检查清单”和“自测”。
- 点击左侧章节后，右侧内容必须跟着切换。
- 点击“检查清单”和“自测”时，必须跳转到对应锚点。
- 页面最终发布到 GitHub Pages。

不要把这个项目改成登录系统、CMS、评论系统、在线编辑器或复杂后台。

## 主要文件

- `app/page.tsx`
  - 页面结构
  - 左侧目录
  - 章节点击切换
  - 移动端目录抽屉
  - `#checklist` / `#self-test` 跳转逻辑

- `app/chapter-data.ts`
  - 所有章节正文数据
  - 每章的标题、卷信息、检查清单、自测内容

- `app/globals.css`
  - 页面样式
  - 阅读排版
  - 侧边栏样式
  - 移动端适配

- `next.config.ts`
  - Next.js 静态导出配置
  - GitHub Pages 路径配置

- `.github/workflows/deploy-pages.yml`
  - GitHub Pages 自动部署配置

## 修改正文时

如果用户只是要改文字，优先修改：

```txt
app/chapter-data.ts
```

不要为了改文字去重写页面结构。

每章结构大致如下：

```ts
{
  id: 16,
  hash: "#ch16",
  title: "...",
  kicker: "...",
  checklist: [],
  selfTest: []
}
```

规则：

- `hash` 必须唯一。
- `checklist` 是“检查清单”。
- `selfTest` 是“自测”。
- 不要额外展示章节正文长段落。
- 删除章节内容时，也要检查目录里是否还残留。

## 修改目录时

如果用户要增加、删除或调整左侧目录，优先修改：

```txt
app/page.tsx
```

重点检查 `parts`。

规则：

- 删除某一卷时，要删除整块卷目录。
- 删除某一章时，要删除对应章节目录项。
- 如果删除了目录项，也要检查 `app/chapter-data.ts` 是否还有对应正文数据。
- 如果删除了正文数据，也要检查 `app/page.tsx` 是否还有对应目录项。
- 目录和正文必须同步，不能只删一边。

## 交互约束

页面交互必须保持：

- 点击章节：切换右侧正文内容。
- 点击章节：URL hash 更新为章节 hash，例如 `#ch16`。
- 当前章节：左侧高亮。
- 当前章节：展开“检查清单”和“自测”两个子项。
- 点击“检查清单”：滚动到 `#checklist`。
- 点击“自测”：滚动到 `#self-test`。
- 移动端点击目录项后：目录抽屉自动关闭。

不要把章节点击改成跳转到一个不存在的新页面。本项目当前是单页阅读体验。

## 样式约束

页面应保持阅读页风格：

- 顶部栏保持 sticky。
- 桌面端保留左侧 sticky sidebar。
- 移动端保留汉堡按钮和抽屉目录。
- 正文最大宽度保持适合阅读，不要铺满全屏。
- section 要保留足够的 `scroll-margin-top`，避免被顶部栏遮住。
- 页面不能出现横向滚动。
- 文字不能重叠、溢出或异常换行。

## 工具和命令

本地开发：

```bash
npm run dev
```

代码检查：

```bash
npm run lint
```

静态构建：

```bash
npm run build
```

搜索残留内容：

```bash
rg "关键词" app out
```

例如删除第九卷后，应检查：

```bash
rg "第九卷|工程模板合集|ch32|Part IX|模板附录" app out
```

## 验收标准

每次改动后，至少确认：

- `npm run lint` 通过。
- `npm run build` 通过。
- 左侧目录可以点击。
- 点击章节后右侧内容会变化。
- “检查清单”能跳到 `#checklist`。
- “自测”能跳到 `#self-test`。
- 移动端目录能打开和关闭。
- 顶部栏不遮挡 section 标题。
- 页面没有横向滚动。
- 已删除的章节或卷不再出现在 `app/` 和 `out/` 中。

如果用户说线上还能看到旧内容，要依次检查：

1. 本地源码。
2. `out/` 构建产物。
3. GitHub 仓库内容。
4. GitHub Pages 线上页面。
5. 浏览器缓存。

## 发布约束

发布目标是 GitHub Pages。

发布前必须先运行：

```bash
npm run lint
npm run build
```

发布时只发布静态构建产物：

```txt
out/
```

不要把 `node_modules/`、`.next/`、本地日志、token 或临时文件发布到线上仓库。

## 安全约束

严禁把以下内容写入仓库：

- GitHub Token
- 密码
- 私钥
- cookie
- 账号凭证
- 任何临时访问密钥

如果用户在聊天里发过 token，使用后要提醒用户去 GitHub 撤销。

## Git 约束

- 不要回滚用户未要求回滚的修改。
- 不要使用 `git reset --hard`。
- 不要使用会破坏用户工作的命令。
- 修改前先看 `git status`。
- 提交前确认只包含本次任务相关文件。

## Agent 工作流程

每次任务按这个顺序做：

1. 判断用户要改的是内容、目录、样式、交互还是部署。
2. 找到对应文件。
3. 小范围修改。
4. 搜索是否有残留。
5. 运行 `npm run lint`。
6. 运行 `npm run build`。
7. 如果涉及线上，发布后检查线上页面。
8. 最后告诉用户改了什么、验证了什么、是否已经发布。

## 不要做的事

- 不要重写整个项目。
- 不要随意引入新的 UI 框架。
- 不要把单页体验改成多页路由。
- 不要删除用户没有要求删除的章节。
- 不要改动和任务无关的文件。
- 不要在没有验证的情况下说“已经好了”。
