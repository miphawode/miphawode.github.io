# 页面仓库 Agent 约束

这个文件是给 AI Agent 看的项目规则。后续修改本项目时，必须优先遵守这里的约束。

## 最高优先级

每次开始干活前，AI Agent 必须先阅读本文件，再执行命令、修改文件或发布项目。

## 项目目标

本项目是一个静态书籍网页：

- 左侧是章节目录。
- 右侧是当前章节内容。
- 每章只展示“检查清单”和“自测”。
- 点击左侧章节后，右侧内容必须跟着切换。
- 点击“检查清单”和“自测”时，必须跳转到对应锚点。
- 目录末尾有“项目笔记”，其中“Git 功能笔记”跳转到 `/editor` 页面。
- `/editor` 是 Git 功能笔记页，支持编辑文本、添加图片和本地保存。
- 页面最终发布到 GitHub Pages。

不要把这个项目改成登录系统、CMS、评论系统、在线编辑器或复杂后台。

## 主要文件

- `app/page.tsx`
  - 页面结构
  - 左侧目录
  - 章节点击切换
  - 移动端目录抽屉
  - `#checklist` / `#self-test` 跳转逻辑
  - “项目笔记 / Git 功能笔记”目录入口

- `app/chapter-data.ts`
  - 所有章节正文数据
  - 每章的标题、卷信息、检查清单、自测内容

- `app/editor/page.tsx`
  - Git 功能笔记页
  - 默认 Git 笔记内容
  - 文本编辑
  - 图片上传和图片链接
  - 浏览器本地保存

- `app/globals.css`
  - 页面样式
  - 阅读排版
  - 侧边栏样式
  - 移动端适配
  - Git 功能笔记编辑页样式

- `next.config.ts`
  - Next.js 静态导出配置
  - GitHub Pages 路径配置

- `.github/workflows/deploy-pages.yml`
  - GitHub Pages 自动部署配置

## 修改正文时

如果用户只是要改文字，优先修改：

```txt
app/chapter-data.ts
```

不要为了改文字去重写页面结构。

每章结构大致如下：

```ts
{
  id: 16,
  hash: "#ch16",
  title: "...",
  kicker: "...",
  checklist: [],
  selfTest: []
}
```

规则：

- `hash` 必须唯一。
- `checklist` 是“检查清单”。
- `selfTest` 是“自测”。
- 不要额外展示章节正文长段落。
- 删除章节内容时，也要检查目录里是否还残留。

## 修改目录时

如果用户要增加、删除或调整左侧目录，优先修改：

```txt
app/page.tsx
```

重点检查 `parts`。

规则：

- 删除某一卷时，要删除整块卷目录。
- 删除某一章时，要删除对应章节目录项。
- 添加普通章节时，使用 `num` 和标题，并确认 `app/chapter-data.ts` 有对应 `hash` 内容。
- 添加独立页面入口时，可以在目录项里使用 `href`，例如 `/editor`。
- 如果删除了目录项，也要检查 `app/chapter-data.ts` 是否还有对应正文数据。
- 如果删除了正文数据，也要检查 `app/page.tsx` 是否还有对应目录项。
- 目录和正文必须同步，不能只删一边；但 `/editor` 这类独立页面不需要写进 `chapter-data.ts`。

## 修改 Git 功能笔记页时

如果用户要改 Git 功能笔记页，优先修改：

```txt
app/editor/page.tsx
```

规则：

- 默认标题在 `DEFAULT_TITLE`。
- 默认正文在 `DEFAULT_BODY`。
- 如果要让线上默认内容发生变化，必须修改 `DEFAULT_BODY`，不能只在浏览器里点“保存”。
- 编辑页里的“保存”只写入当前浏览器的 `localStorage`，不会自动提交到 Git，也不会影响其他设备。
- 如果用户说线上默认 Git 笔记没变，要提醒他清空浏览器本地草稿，或者确认是否真的改了 `DEFAULT_BODY`。
- 图片上传会转成浏览器本地数据，只适合个人草稿；如果要把图片作为长期线上内容，应放到 `public/` 或使用稳定图片 URL。
- 不要把 GitHub Token、cookie、私密截图上传进编辑页默认内容。

## 交互约束

页面交互必须保持：

- 点击章节：切换右侧正文内容。
- 点击章节：URL hash 更新为章节 hash，例如 `#ch16`。
- 当前章节：左侧高亮。
- 当前章节：展开“检查清单”和“自测”两个子项。
- 点击“检查清单”：滚动到 `#checklist`。
- 点击“自测”：滚动到 `#self-test`。
- 点击“项目笔记 / Git 功能笔记”：跳转到 `/editor` 页面。
- 点击顶部“Git 笔记”：跳转到 `/editor` 页面。
- 在 `/editor` 点击“阅读页”：返回 `/`。
- 移动端点击目录项后：目录抽屉自动关闭。

不要把章节点击改成跳转到一个不存在的新页面。章节阅读区仍是单页体验；只有 `/editor` 这类独立项目笔记页使用真实路由。

## 样式约束

页面应保持阅读页风格：

- 顶部栏保持 sticky。
- 桌面端保留左侧 sticky sidebar。
- 移动端保留汉堡按钮和抽屉目录。
- 正文最大宽度保持适合阅读，不要铺满全屏。
- section 要保留足够的 `scroll-margin-top`，避免被顶部栏遮住。
- 页面不能出现横向滚动。
- 文字不能重叠、溢出或异常换行。
- `/editor` 页面要保持左右分栏：左侧编辑，右侧预览。
- `/editor` 移动端要退化为上下布局，不能让文本框或图片列表撑出横向滚动。
- `/editor` 图片预览必须限制宽度和高度，不能把页面撑破。

## 工具和命令

本地开发：

```bash
npm run dev
```

代码检查：

```bash
npm run lint
```

静态构建：

```bash
npm run build
```

搜索残留内容：

```bash
rg "关键词" app out
```

例如删除第九卷后，应检查：

```bash
rg "第九卷|工程模板合集|ch32|Part IX|模板附录" app out
```

检查 Git 笔记入口和静态导出：

```bash
rg "Git 功能笔记|项目笔记|/editor" app out
```

## 验收标准

每次改动后，至少确认：

- `npm run lint` 通过。
- `npm run build` 通过。
- 左侧目录可以点击。
- 点击章节后右侧内容会变化。
- “检查清单”能跳到 `#checklist`。
- “自测”能跳到 `#self-test`。
- 移动端目录能打开和关闭。
- 顶部栏不遮挡 section 标题。
- 页面没有横向滚动。
- 已删除的章节或卷不再出现在 `app/` 和 `out/` 中。
- 如果改了 Git 功能笔记页，`/editor` 必须能静态导出为 `out/editor/index.html`。
- 如果改了目录入口，首页 `out/index.html` 中必须能搜到“项目笔记”“Git 功能笔记”和 `/editor`。
- 如果改了编辑页默认正文，要确认 `out/editor/index.html` 中能搜到新增文本。

如果用户说线上还能看到旧内容，要依次检查：

1. 本地源码。
2. `out/` 构建产物。
3. GitHub 仓库内容。
4. GitHub Pages 线上页面。
5. 浏览器缓存。
6. `/editor` 的浏览器本地草稿是否覆盖了新的默认内容。

## 发布约束

发布目标是 GitHub Pages。

发布前必须先运行：

```bash
npm run lint
npm run build
```

发布时只发布静态构建产物：

```txt
out/
```

不要把 `node_modules/`、`.next/`、本地日志、token 或临时文件发布到线上仓库。

GitHub Pages 必须保持使用 `workflow` 发布方式。不要把 Pages 切回从 `master` 分支根目录发布，因为仓库根目录是源码，不是最终静态网页。

## 安全约束

严禁把以下内容写入仓库：

- GitHub Token
- 密码
- 私钥
- cookie
- 账号凭证
- 任何临时访问密钥

如果用户在聊天里发过 token，使用后要提醒用户去 GitHub 撤销。

## Git 约束

- 不要回滚用户未要求回滚的修改。
- 不要使用 `git reset --hard`。
- 不要使用会破坏用户工作的命令。
- 修改前先看 `git status`。
- 提交前确认只包含本次任务相关文件。
- 如果只发布 Git 笔记页和目录入口，优先提交：

```bash
git add app/page.tsx app/globals.css app/editor/page.tsx
```

- 如果 `app/chapter-data.ts` 还有用户正在编辑的答案，除非用户明确要求，不要顺手提交进去。

## Agent 工作流程

每次任务按这个顺序做：

1. 先阅读 `agent.md`。
2. 判断用户要改的是内容、目录、样式、交互还是部署。
3. 找到对应文件。
4. 小范围修改。
5. 如果涉及 `/editor`，确认是要改线上默认内容，还是只教用户在浏览器里临时编辑。
6. 搜索是否有残留或入口是否存在。
7. 运行 `npm run lint`。
8. 运行 `npm run build`。
9. 如果涉及线上，发布后检查线上页面。
10. 最后告诉用户改了什么、验证了什么、是否已经发布。

## 不要做的事

- 不要重写整个项目。
- 不要随意引入新的 UI 框架。
- 不要把单页体验改成多页路由。
- 不要删除用户没有要求删除的章节。
- 不要改动和任务无关的文件。
- 不要在没有验证的情况下说“已经好了”。
- 不要把浏览器 localStorage 当作已经发布到 GitHub 的内容。
